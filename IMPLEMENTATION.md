# מסמך מימוש

## תיאור קצר
זהו פרויקט React + Vite עם API פשוט של Vercel Serverless.

הממשק הראשי נמצא ב־`src/App.jsx`, והשרת נמצא ב־`api/progress.js`.

## מבנה הפרויקט
- `src/App.jsx` - כל הלוגיקה והתצוגה של דף המעקב
- `api/progress.js` - API לקריאה, שמירה ואיפוס התקדמות
- `vite.config.js` - הגדרות Vite כולל proxy לפיתוח
- `package.json` - סקריפטים לפיתוח ובנייה
- `nitro.config.js` - נשאר מהניסיונות הקודמים, אבל ה־API הפעיל בפרודקשן הוא `api/progress.js`

## מודל הנתונים
ההיסטוריה נשמרת כמערך מסודר של רשומות:

```json
[
  { "day": 0, "value": 2 },
  { "day": 5, "value": 6 }
]
```

פירוש השדות:
- `day` = מספר הימים שעברו מאז `2026-03-10`
- `value` = מספר רבעי העמוד שהושלמו מאז נקודת ההתחלה

## חישוב התקדמות
הקוד מגדיר ארבע יחידות לכל דף:
- `a-top`
- `a-bottom`
- `b-top`
- `b-bottom`

הפונקציה `toIndex(daf, part)` ממירה דף + חלק למספר רציף.

הפונקציה `toPages(parts)` ממירה רבעי עמוד לדפים.

הפונקציה `formatPages(parts, fractionDigits)` מחזירה טקסט מוכן לתצוגה, למשל `1.25 דפים`.

## גבולות התכנית
ב־`src/App.jsx` מוגדרים:
- `startDate = 2026-03-10`
- `targetDate = 2026-11-02`
- סוף המסכת לחישוב: `toIndex(113, "b-bottom")`

שימו לב: החישוב כולו מניח שהתחלת המעקב היא מהנקודה הזאת. אם משנים נקודת התחלה או סיום, צריך לעדכן את הקבועים ואת המשמעות של `value`.

## טעינת נתונים
בעת עליית האפליקציה:
- מתבצעת קריאת `GET /api/progress`
- אם התשובה תקינה, `history` נטען ל־state
- אם לא, מוצגת הודעת `טעינת הנתונים נכשלה`

הטעינה מתבצעת מתוך `useEffect` בתחילת `src/App.jsx`.

## שמירה
הפעולה `save()` ב־`src/App.jsx`:
- בודקת שיש מצב עריכה
- ממירה את התאריך ל־`day`
- ממירה את הדף והחלק ל־`value`
- שולחת `POST /api/progress`
- מוסיפה header בשם `x-edit-key`
- לאחר הצלחה, מחליפה את `history` בנתונים שחזרו מהשרת

בשרת:
- נבדקת התאמה בין `x-edit-key` לבין `EDIT_KEY`
- אם יש כבר רשומה לאותו `day`, היא מוחלפת
- ההיסטוריה נשמרת מחדש כשהיא ממוינת לפי יום

## איפוס
הפעולה `reset()` זמינה רק אם:
- יש מצב עריכה
- המשתמש הגיע עם `#danger-reset`
- המשתמש הקליד ידנית את טקסט האישור

השרת מוחק הכול על ידי `DELETE /api/progress`.

## חישובים במסך
ב־`src/App.jsx` מחושבים בין השאר:
- `percent` - אחוז התקדמות מתוך כל המסכת המנוטרת
- `completedParts` - כמה רבעים הושלמו
- `remainingParts` - כמה רבעים נותרו
- `expectedToday` - כמה היה צריך להשלים עד היום לפי קו ישר
- `gap` - פער מול התכנית
- `forecast` - תחזית סיום לפי הקצב מההיסטוריה בפועל
- `chartData` - הנתונים שמוזנים לגרף
- `milestones` - אבני דרך מרכזיות

## הגרף
הגרף משתמש ב־`recharts`.

הקווים:
- `planned` - הקו הכחול של התכנית
- `actual` - הקו הירוק של ההתקדמות בפועל
- `forecast` - קו תחזית אפור מקווקו

הקו הירוק לא אמור להמשיך אחרי הנקודה האחרונה שיש בה נתון אמיתי.

## ימי לימוד
ה־heatmap של ימי הלימוד נבנה מתוך כל הימים בטווח התכנית.

לכל יום:
- נלקח הערך הידוע האחרון עד אותו יום
- אם יש רשומה לאותו יום, היום מסומן כיום לימוד
- השורות מקובצות לפי חודש
- tooltip מוצג מיידית על hover/focus

## מצב עריכה
ה־frontend קובע מצב עריכה על פי `editKey` מה־URL:
- query param: `?editKey=...`

חשוב:
- זה רק פותח את ממשק העריכה
- האישור האמיתי מתבצע בשרת על בסיס `x-edit-key`

## אחסון בשרת
`api/progress.js` עובד כך:
- אם קיימים `UPSTASH_REDIS_REST_URL` ו־`UPSTASH_REDIS_REST_TOKEN`, משתמש ב־Upstash Redis
- אם קיימים `KV_REST_API_URL` ו־`KV_REST_API_TOKEN`, משתמש בהם גם כן
- אם אין כלל חיבור ענן ובסביבה מקומית, נשמר בזיכרון (`memoryHistory`)

## משתני סביבה חשובים
Production:
- `EDIT_KEY`
- `UPSTASH_REDIS_REST_URL` או `KV_REST_API_URL`
- `UPSTASH_REDIS_REST_TOKEN` או `KV_REST_API_TOKEN`

Local development:
- אם אין `EDIT_KEY`, מצב פיתוח משתמש ב־`dev-edit-key`
- אם אין Redis, הנתונים נשמרים רק בזיכרון של תהליך השרת

## פיתוח מקומי
סקריפטים חשובים:

```bash
npm run dev
```
מריץ frontend ו־API מקומי יחד.

```bash
npm run dev:web:remote
```
מריץ frontend מקומי, אבל מביא נתונים מהאתר המרוחק דרך proxy.

## הערות חשובות למי שממשיך את הפרויקט
- רוב הלוגיקה נמצאת כרגע בקובץ אחד: `src/App.jsx`
- זה נוח לפרויקט קטן, אבל אם המערכת תגדל כדאי לפצל ל־components ו־helpers
- ה־API intentionally פשוט מאוד: endpoint יחיד עם `GET/POST/DELETE`
- אם יש בעיית `Unauthorized`, צריך לבדוק קודם את `EDIT_KEY`, אחר כך את ה־URL, ואז את סביבת Vercel
- אם יש בעיית `טעינת הנתונים נכשלה`, צריך לבדוק קודם את `GET /api/progress`
- בפרודקשן ה־API שעובד הוא זה שבתיקיית `api`, לא ב־`server/api`

- `REQUIREMENTS.md`
- `IMPLEMENTATION.md`
- `src/App.jsx`
- `api/progress.js`

כך הוא יבין גם את הכוונה המוצרית וגם את המימוש בפועל.
